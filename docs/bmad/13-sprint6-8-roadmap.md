# Sprint 6-8 路线图

## 总结

当前产品进入棕地收口阶段。后续不盲目扩展教官端、装备场地、AAR 工作流或成就体系，而是围绕主链路逐步推进：

`稳定 V3.2 -> 自动晋级闭环 -> 后台可读性 -> 部署交付`

## Sprint 6：V3.2 稳定化与自动晋级

要解决的问题：

V3.2 已具备更强的题库体验，但存在答题页运行时错误、答题记录重复写入、题库状态统计失效和晋级闭环未收口的问题。

关键工作：

- 修复 V3.2 答题页退出、标记、答题卡、跳题按钮。
- 修复题组提交，只写入一次正确的题目记录。
- 题库列表使用最新答题记录展示已答、正确、错误状态。
- 知识域筛选作为一级入口展示。
- 完成自动晋级：阶段课程完成 + 阶段课程题正确率大于 90% 后自动更新等级。
- Profile、Today、后台详情展示晋级进度、缺失课程、题目正确率和晋级历史。
- 主入口从 V3.1 切换到 V3.2。

验收：

- V3.2 主路径可完成注册、学习、答题、Profile、后台查看。
- 90% 正确率不晋级，91% 正确率自动晋级。
- `level_history` 有晋级记录。
- `npm run check`、`npm run test:api`、`npm run test:perf` 通过。

## Sprint 7：后台运营可读性增强

要解决的问题：

后台可以看到数据，但还需要更快判断哪些用户需要关注，以及为什么需要关注。

关键工作：

- 增加只读筛选：风险状态、最近活动、弱项能力域、等级。
- 增加运营分组：未开始用户、中断用户、低正确率用户、高进度用户、可晋级/刚晋级用户。
- 用户详情增强：课程进度时间线、题目表现趋势、强项/弱项 TOP、下一步建议。
- 保持后台只读，不新增编辑、删除、重置。

验收：

- 管理员 1 分钟内能判断需要关注的用户。
- 用户详情能解释风险标签和晋级卡点。
- 移动端后台仍可读，桌面端信息密度提高。

## Sprint 8：部署交付与版本治理

要解决的问题：

产品可以本地演示，但需要更清晰的交付、备份、版本和质量门禁。

关键工作：

- 固化本地部署说明：Node 版本、启动命令、端口、数据目录、SQLite 备份恢复。
- 固化版本治理：V3.2 为主学员端，admin.html 为后台端，旧版本只保留参考。
- 强化质量门禁：主入口、题库三入口、自动晋级、后台只读、V3.2 不显示课程图片。
- 准备小范围试用交付说明。

验收：

- 新开发者能按文档启动和检查。
- 演示人员能按文档完成五分钟演示。
- 数据备份和恢复路径明确。
- 检查脚本能发现入口或晋级规则漂移。
