# app2 文档差距矩阵

## 1. 需求文档

| 模块 | 当前材料 | 缺口 | 建议补齐内容 |
|---|---|---|---|
| 项目目标 | 改版规格总纲已有定位 | 缺少正式 PRD 结构 | 写明产品目标、非目标、版本范围、成功指标。 |
| 用户角色 | 代码有 member/instructor；资料有队员/骨干/教官 | 角色口径不统一 | 定义队员、班长、骨干、教官、管理员的权限和使用场景。 |
| 核心流程 | 代码已有学习和答题流程 | 缺少端到端业务流程 | 补“课前简报-训练执行-题库检验-AAR-补训-晋升”的流程图。 |
| 功能清单 | 功能新增文档有 P0/P1/V2 | 未和当前实现逐项对齐 | 建立功能状态表：已完成、部分完成、未开始、暂缓。 |
| 验收标准 | 改版规格有内容/功能/技术验收 | 缺少可执行验收用例 | 每项验收补输入、操作步骤、预期结果、负责角色。 |

最低交付物：`docs/requirements.md`。

## 2. 设计文档

| 模块 | 当前材料 | 缺口 | 建议补齐内容 |
|---|---|---|---|
| 信息架构 | H5 已有首页/课程/题库/战绩 | 未说明页面层级和跳转 | 补站点地图、导航规则、空状态和异常状态。 |
| 队员端设计 | `public/app.js` 中直接实现 | 缺少用户任务视角 | 明确“今日任务、课程详情、答题、档案”的页面目标和主操作。 |
| 教官端设计 | `public/admin.js` 中直接实现 | 只看总览，不能管理 | 补队员详情、班组视图、AAR 审阅、补训建议、评估录入。 |
| 交互状态 | 代码中隐含 | 缺少状态表 | 补登录态、加载、提交中、错误、无数据、已完成、已解锁等状态。 |
| 视觉规范 | CSS 已有主题变量 | 缺少设计系统说明 | 补颜色、字号、间距、组件、移动端断点和可访问性要求。 |

最低交付物：`docs/design.md`。

## 3. 数据库文档

| 模块 | 当前材料 | 缺口 | 建议补齐内容 |
|---|---|---|---|
| 用户 | `data/users.json` | 无正式字段说明 | 定义用户表字段、角色枚举、手机号唯一约束、密码策略。 |
| 进度 | `data/progress.json` | 模型嵌套但无 schema | 定义 lessons、quizzes、questionAttempts、tagStats、achievements。 |
| AAR | `data/aar.json` | 后端有接口但 UI 不完整 | 定义 AAR 记录字段、弱项标签、可见范围、保留策略。 |
| 评估 | `data/assessments.json` | 权限和对象不清 | 定义评估对象、评分维度、教官权限、班组关联。 |
| 课程内容 | `public/training-data.js` | 静态 JS 数据，不利于校验 | 定义 course/week/lesson/question/tag/achievement 的结构和校验规则。 |
| 迁移路线 | 无 | JSON 不适合正式多人使用 | 建议迁移 SQLite：users、sessions、weeks、lessons、questions、progress、aar、assessments、achievements。 |

最低交付物：`docs/database.md`，并附 JSON 到 SQLite 的迁移策略。

## 4. 技术文档

| 模块 | 当前材料 | 缺口 | 建议补齐内容 |
|---|---|---|---|
| 架构 | 从代码可读出 | 无正式说明 | 补运行架构、目录结构、请求链路、静态资源策略。 |
| API | 代码内隐含 | 无接口文档 | 补 `/api/login`、`/api/register`、`/api/me`、`/api/progress`、`/api/aar`、`/api/assessment`、`/api/admin/users` 的契约。 |
| 安全 | 代码有基础密码哈希 | 无安全策略 | 补认证、授权、Token 过期、密码重置、默认账号禁用、数据脱敏。 |
| 部署 | 只有 `npm start` | 无环境说明 | 补 Node 版本、端口、数据目录、反向代理、备份恢复。 |
| 测试 | 无 | 缺少质量门禁 | 补语法检查、数据校验、API 冒烟、权限测试、UI 主路径测试。 |
| 运维 | 有日志文件但未规范 | 无监控与备份 | 补日志、错误排查、数据备份、恢复演练、版本升级。 |

最低交付物：`docs/technical.md`。

## 5. 推荐文档推进顺序

1. 先写 `requirements.md`，锁定 MVP 范围和 L0-L4/L0-L6 口径。
2. 再写 `database.md`，把课程、题库、进度、AAR、评估的数据契约稳定下来。
3. 接着写 `design.md`，按稳定的数据和流程设计队员端/教官端。
4. 最后写 `technical.md`，定义实现、部署、测试和运维门禁。
