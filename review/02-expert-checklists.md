# app2 专家清单式评审

## 1. 业务评审

| 优先级 | 检查项 | 结论 | 建议 |
|---|---|---|---|
| P0 | 30 周主线是否完整 | 通过 | 已有 6 阶段、30 周、90 子课，可作为课程骨架。 |
| P0 | 课程是否达到训练组织深度 | 不通过 | 补齐任务简报、里程碑、深度拓展、标靶训练卡。 |
| P0 | 题库是否足以检验学习效果 | 不通过 | 按每周、每技能域补齐情景题和课程题。 |
| P0 | 教官/队员核心流程是否闭环 | 部分通过 | 队员可学习和答题；教官缺少任务下发、AAR 管理、评估闭环。 |
| P1 | 等级与晋升是否清晰 | 不通过 | 统一根目录资料的 L0-L4 与改版规格的 L0-L6。 |
| P1 | 训练后复盘是否可沉淀 | 部分通过 | 后端有 AAR 接口，但主界面未形成稳定入口和管理流程。 |

业务结论：课程骨架成立，但还不是完整训练产品。当前更像“训练内容浏览 + 答题原型”，距离“训练组织系统”仍缺关键业务能力。

## 2. UX 评审

| 优先级 | 检查项 | 结论 | 建议 |
|---|---|---|---|
| P0 | 移动端主路径是否清楚 | 部分通过 | 底部导航清楚，但首页应强化“今日训练任务”。 |
| P0 | 学习反馈是否即时 | 通过 | 答题后有判定、解析、标签统计和成就反馈。 |
| P1 | 课程详情是否好用 | 不通过 | 目前信息偏少，无法支持教官现场组织训练。 |
| P1 | 管理端是否可行动 | 部分通过 | 管理端有总览，但缺少队员详情、班组视角和补训建议。 |
| P1 | 中文体验是否稳定 | 通过 | 文件为 UTF-8，应用文本正常；PowerShell 默认编码读取会乱码，但不是应用问题。 |
| P2 | 视觉风格是否符合训练主题 | 部分通过 | 暗色战术风格统一，但信息密度与长期阅读舒适度需要实测。 |

UX 结论：视觉方向基本成立，队员答题体验比课程学习体验更完整。下一步应把“今天该做什么、做完怎么记录、教官如何处理结果”放到首屏核心路径。

## 3. 技术评审

| 优先级 | 检查项 | 结论 | 建议 |
|---|---|---|---|
| P0 | JS 语法检查 | 通过 | `server.js`、`public/app.js` 均通过 `node --check`。 |
| P0 | API 鉴权 | 部分通过 | 需要登录 Token；管理用户列表有 instructor 权限检查。 |
| P0 | 会话管理 | 不通过 | Token 无过期、无服务端退出、无撤销、重启丢失。 |
| P0 | 持久化可靠性 | 不通过 | JSON 文件直接覆盖写，存在并发写丢数据风险。 |
| P0 | 错误处理 | 部分通过 | API 有基本错误返回，但所有异常统一 400，不利于定位。 |
| P1 | 前端状态管理 | 部分通过 | 单文件状态可维护性有限，后续功能增加会快速膨胀。 |
| P1 | API 结构 | 部分通过 | 接口数量少且清楚，但缺少版本、契约文档和输入校验细则。 |
| P1 | 测试体系 | 不通过 | `package.json` 只有 `start`，无测试、lint、构建、数据校验脚本。 |

技术结论：原型实现务实，但正式化前必须处理会话、持久化、数据校验和测试脚本。否则用户规模稍扩大就会出现数据可信度问题。

## 4. 数据评审

| 优先级 | 检查项 | 结论 | 建议 |
|---|---|---|---|
| P0 | 课程数据数量 | 通过 | 30 周、90 子课。 |
| P0 | 题目 ID 唯一性 | 通过 | 当前未发现重复题目 ID。 |
| P0 | 规格字段完整性 | 不通过 | `missionBriefing`、`rangeTraining`、`deepDives`、`milestoneChecklist` 全部缺失。 |
| P0 | 题库覆盖 | 不通过 | 情景题 20、课程题 25，低于规格。 |
| P1 | 标签体系 | 部分通过 | 有 45 个标签，但缺少标签字典、技能域归类和命名规范。 |
| P1 | 进度模型 | 部分通过 | 能记录完成和答题，但缺少阶段验收、教官确认、补训闭环。 |
| P1 | 评估模型 | 部分通过 | assessment 有评分维度，但没有正式数据库文档和前端录入流程。 |

数据结论：课程与题库数据已经能驱动演示，但不具备正式训练档案所需的完整性、可追溯性和统计可信度。

## 5. 安全评审

| 优先级 | 检查项 | 结论 | 建议 |
|---|---|---|---|
| P0 | 默认管理员凭据 | 不通过 | 不应在登录页和日志中暴露默认账号密码。 |
| P0 | 密码存储 | 通过 | 使用 PBKDF2 + salt，方向正确。 |
| P0 | Token 存储 | 部分通过 | 前端存在 localStorage；后端内存 Map，无过期策略。 |
| P0 | 管理接口权限 | 部分通过 | `/api/admin/users` 有角色校验；`/api/assessment` 读取未区分普通用户和教官。 |
| P1 | 输入限制 | 部分通过 | body 有 1MB 限制，字段长度和数组内容仍需更严格校验。 |
| P1 | 数据隐私 | 不通过 | 管理端展示手机号，缺少脱敏策略和访问审计。 |

安全结论：本地原型可接受，真实队伍使用前必须移除默认凭据暴露、补会话过期、完善接口权限和手机号脱敏。
