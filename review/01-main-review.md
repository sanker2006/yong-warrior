# app2 主评审报告

## 1. 结论

`app2` 已具备可演示的训练学习闭环：登录/注册、30 周课程地图、周训练详情、题库训练、标签画像、成就展示、管理员总览。它适合作为内部原型或早期试用版本。

但按现有《改版规格》与训练总纲要求，当前版本尚未达到可正式推进 MVP 的标准。核心原因是：课程内容深度不足、题库数量不足、任务简报/标靶训练/里程碑/深度拓展等规格字段缺失，且安全、数据持久化、管理员能力仍偏原型化。

阶段建议：先进入“文档补齐 + P0 修复”阶段，不建议直接上线给完整队伍长期使用。

## 2. 当前实现概览

| 维度 | 现状 |
|---|---|
| 技术栈 | Node 原生 HTTP 服务 + 静态 HTML/CSS/JS |
| 持久化 | `data/*.json` 文件存储 |
| 用户角色 | `member`、`instructor` |
| 队员端 | 首页、课程、题库、战绩与成就 |
| 管理端 | 用户列表、周进度、题库正确率、标签强弱项、成就数量 |
| 课程数据 | 6 阶段、30 周、90 个子课 |
| 题库数据 | 45 题，其中情景题 20、课程题 25 |
| 成就体系 | 5 个成就 |

## 4. P0 阻塞项

1. 规格核心字段未落地。  
   `missionBriefing`、`deepDives`、`milestoneChecklist`、`rangeTraining` 全部缺失，导致课程详情仍是“周概览 + 子课摘要”，没有达到训练组织和复盘所需的内容层级。

2. 题库规模不满足验收标准。  
   当前 45 题距离规格要求的 180 题总量差距较大；情景题少 40 道，课程题少 95 道。题库无法覆盖 30 周训练目标，也难以支撑标签画像的可信度。

3. 用户/角色管理仍偏原型。  
   注册用户默认都是 `member`，缺少教官创建、角色调整、队员分组、禁用账号、重置密码等管理动作。现有管理端只能看结果，不能运营训练过程。

4. 默认管理员账号直接展示。  
   登录页和服务启动日志暴露默认账号密码，适合本地调试，不适合真实队伍使用。

5. 会话无过期与撤销机制。  
   Token 只存内存 `Map`，服务重启全部失效；同时没有过期时间校验、退出服务端失效、设备管理或强制下线。

6. JSON 文件存储缺少并发保护。  
   `readJson` + `writeJson` 直接读写文件，在多人同时答题或提交记录时可能出现覆盖写。正式使用前至少需要写入队列/文件锁，或迁移到 SQLite。

## 5. P1 体验与运营问题

1. 学习路径没有“今日任务”视角。  
   当前首页显示下一训练周，但没有把“教官今天要组织什么、队员今天要完成什么、训练后要提交什么”拆成明确任务。

2. 课程详情偏摘要。  
   每周只有目标、标准、警示和 3 个子课，缺少规格中要求的任务简报、标靶训练卡、里程碑自检、参考资料、深度拓展。

3. 管理端缺少可行动建议。  
   管理端能看到薄弱标签，但没有按班组、阶段、周次生成补训建议，也不能下发任务或查看单个队员详情。

4. 等级体系与训练资料不一致。  
   根目录团队资料定义 L0-L4；改版功能文档定义 L0-L6；代码只保存 `level` 字段但没有晋升规则。需要先统一口径。

5. AAR 与 assessment 接口未进入主 UI。  
   服务端已有 `/api/aar`、`/api/assessment`，但队员端和管理端没有完整录入/查看流程，训练闭环没有真正用起来。

## 6. V2 后续增强

- 高级数据分析、个性化训练推荐、导出 PDF/Excel、回放/观战、移动端原生 App 可放入 V2。
- 社交、排行榜、电商、AI 对手暂不建议进入近期范围，容易偏离训练主线。

## 7. 是否可进入下一阶段

建议进入下一阶段，但下一阶段应定义为“P0 修复与正式文档补齐”，不是正式发布。

进入正式 MVP 发布前，最低门槛如下：

- 补齐正式需求文档、设计文档、数据库文档、技术文档。
- 明确 L0-L6 或 L0-L4 的最终等级体系，只保留一种口径。
- 补齐 `missionBriefing`、`milestoneChecklist`、`deepDives`，并实现页面展示。
- 题库至少达到 MVP 可验收量，并覆盖每阶段关键技能。
- 移除默认账号暴露，补充会话过期和基础权限管理。
- 将 JSON 文件存储升级为带并发保护的方案，推荐 SQLite。
